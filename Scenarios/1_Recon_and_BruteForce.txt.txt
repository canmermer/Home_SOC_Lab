### 1 — Recon and Brute-Force (Windows Firewall Enabled)
In this step, I scanned the target while Windows firewall was turned on to see our baseline security posture.

* **Commands Executed on Kali:**
  ```bash
  sudo nmap -sV -p- 192.168.56.20

Command Breakdown:

    -p- : Tells Nmap to scan all 65,535 possible TCP ports instead of just the top 1,000 common ones.

    -sV : Enables service version detection, which tries to figure out exactly what software version is running on any open port it finds.

Results:

The Windows Firewall dropped the probes on almost every port, showing 65,533 ports as filtered (meaning they gave no response). However, because OpenSSH is running on the target, Port 22 was discovered wide open:

Results and Analysis:

    Tool Behavior: Hydra stopped after 3 attempts with a connection error. Medusa failed immediately because it could not connect to the SSH service.

    Windows Defense: The target Windows OS detected the fast attack. Instead of a firewall block, Windows dropped the connection pool for the attacker IP (192.168.56.30) to protect the system.

    SIEM Check: In Wazuh, I found multiple Rule 60122 alerts (Windows Event ID 4625 - Invalid password). The SSH service did not crash, but it blocked the traffic.

1.C — Service Reset & Operator Login

To fix the connection block and test a normal login, I restarted the service.

    Command on Windows Target (PowerShell):
    PowerShell

    Restart-Service sshd

SIEM Logs:

    Service Restart: The restart cleared the connection block. Wazuh triggered Rule 61104 (Service startup type/state changed) at 14:31:23.

    Operator Connection: The admin connected from the host machine at 14:32:05. The system showed a "Potential Security Risk" warning because the service keys changed.

    Legitimate Log: The admin accepted the warning, entered the password, and generated a Windows Logon Success (Rule 60106) log at 14:32:13.089.
